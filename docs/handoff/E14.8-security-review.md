## 1. Findings

| # | Severity | OWASP | Где | Статус |
|---|---|---|---|---|
| **F1** | Medium | A05/A06 | `JwksKeyResolver` — JWKS-fetch без connect/read timeout | ✅ исправлено |
| **F2** | Medium | A04 | `JwksKeyResolver` — нет negative-cache по неизвестному `kid` (pre-auth amplification) | ✅ исправлено |
| **F3** | Medium | A04 | Нет лимита размера тела; неаутентиф. OM-webhook буферизует unbounded `byte[]` | ✅ исправлено |
| **F4** | Medium | A10 | `SubscriptionService` — outbound webhook URL без SSRF-guard (только scheme-check) | 📋 задокументировано (§3) |
| **F5** | Medium | A05 | CSP / security-заголовки не настроены нигде | ✅ частично (baseline+doc) |
| F-UI-1 | Low | — | StrictMode → двойной OIDC code-exchange («Code not valid») | ✅ исправлено (ранее в сессии) |

### F1 — JWKS-fetch без таймаутов

`JwksKeyResolver` строил `new UrlJwkProvider(jwksUrl)` без connect/read
timeout'ов. JWT-валидация — на пути **каждого** запроса; висящий/медленный
Keycloak бесконечно держал бы Jersey-request-thread → availability-DoS.

**Фикс.** Конструктор `UrlJwkProvider(URL, connectTimeout=2000, readTimeout=3000)`
(API стабилен в jwks-rsa 0.22.1; подтверждено компиляцией).

### F2 — нет negative-cache по неизвестному `kid`

Проверка подписи идёт **до** любой авторизации. Неаутентифицированный клиент,
подставляя случайные `kid`, заставлял сервис на КАЖДЫЙ запрос ходить в
Keycloak за полным JWKS (amplification + thread-pressure: успехи кэшировались,
промахи — нет).

**Фикс.** Добавлен Caffeine negative-cache (TTL 30 c, max 256): `kid`,
по которому `delegate.get` бросил `JwkException`, кэшируется → повторный
bogus-`kid` не порождает сетевой fetch. Короткий TTL не мешает штатной
key-rotation.

### F3 — нет лимита размера тела запроса

Подтверждено: лимита нет нигде (`server:`-блок `config.yml`, Jetty,
фильтры). `OwnershipWebhookResource.receive(byte[] rawBody)` —
**единственный неаутентифицированный POST** — буферизует тело целиком в heap
**до** HMAC-проверки (HMAC обязан считаться по сырым байтам). Multi-GB-тело
от неаутентифицированного клиента → OOM.

**Фикс.** `bank.rdmmesh.app.security.RequestSizeLimitFilter`
(`@Provider @PreMatching @Priority(1)`) — отбивает `Content-Length >
1 MiB` с `413` до resource-матчинга/auth/чтения entity. Зарегистрирован
в `RdmmeshApplication.run()` первым. **Граница:** проверяется
`Content-Length` (OM Event Subscription и обычные клиенты его шлют);
chunked-without-length не покрыт — см. §3.

### F4 — SSRF в outbound webhook (задокументировано, не исправлено)

`SubscriptionService.create` валидирует только схему `http(s)`. URL
admin-supplied; `WebhookDeliveryWorker` POST'ит туда подписанные
published-payload'ы. Митигаторы present: endpoint admin-only
(`@RolesAllowed("RDM_ADMIN")`), `HttpClient.followRedirects(NEVER)`
(закрывает redirect-bypass). Остаточный риск: over-privileged/обманутый
admin → сервис POST'ит во внутреннюю сеть / `169.254.169.254` / admin-порт
`:8081`.

**Почему не исправлено в этом round'е.** Корректный SSRF-guard — это
allow/deny-list резолвленных адресов (loopback/link-local/private/metadata)
+ DNS-rebinding-safe проверка на момент connect. Без политики банка «какие
сети — легитимные consumer'ы» строгий фильтр сломает реальные внутренние
подписки. Это network-policy decision — см. §3 #1 и §6 Q56.

### F5 — CSP / security-заголовки

Подтверждено: `Content-Security-Policy`/`X-Frame-Options`/HSTS не настроены
ни в `index.html`, ни в Vite, ни на backend'е (UI в prod — за
nginx/ingress, конфиг которого в репо нет — `helm/` заглушка). Активного
XSS-sink нет (см. §0), но отсутствует defense-in-depth + анти-clickjacking.

**Фикс (частичный, env-агностичный).** В `index.html` добавлен baseline
`<meta http-equiv="Content-Security-Policy">` =
`object-src 'none'; base-uri 'self'; form-action 'self'` — только директивы
без script/connect/style-импликаций, безопасны и в Vite-dev, и в prod,
не ломают OIDC (redirect — top-level navigation). Полный prod-CSP — на
reverse-proxy, см. §3 #2 (шаблон).

## 3. Что осталось — follow-up'ы

1. **F4 SSRF-guard (Round 9 или security-follow-up).** Реализовать
   конфигурируемый denylist резолвленных адресатов в `WebhookDeliveryWorker`
   (на момент connect, DNS-rebinding-safe): по умолчанию блокировать
   loopback / link-local (`169.254/16`, `fe80::/10`) / `127/8` / `::1` /
   private (`10/8`,`172.16/12`,`192.168/16`) / cloud-metadata. Override —
   allowlist из конфига под политику банка. Нужен decision §6 Q56.
2. **Прод-CSP/заголовки на reverse-proxy (V1 deploy).** `frame-ancestors`
   через `<meta>` спекой игнорируется — только заголовок. Целевой набор для
   ingress/nginx:
   ```
   Content-Security-Policy: default-src 'self';
     script-src 'self' 'nonce-<per-request>';
     style-src 'self' 'unsafe-inline';            # AntD CSS-in-JS
     connect-src 'self' https://<keycloak-host>;  # OIDC token/JWKS
     img-src 'self' data:; object-src 'none'; base-uri 'self';
     form-action 'self'; frame-ancestors 'none'
   Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
   X-Content-Type-Options: nosniff
   Referrer-Policy: no-referrer
   ```
   Положить в `helm/` (сейчас заглушка) при первом prod-деплое.
3. **F3 chunked-without-Content-Length.** Закрыть Jetty-уровнем:
   `httpConfiguration` с `requestHeaderSize`/connector-лимитом на тело,
   либо обёртка `InputStream` с hard-cap. Bundle с Round 9 (testcontainers
   IT — заодно IT на F1/F2/F3).
4. **Тесты на security-фиксы (Round 9).** Юнит/IT: F2 negative-cache
   (bogus-`kid` → один fetch, не N); F3 (413 на oversize Content-Length);
   F1 (timeout не висит). Сейчас покрыто только ручным smoke §5 —
   тот же класс долга, что E14.7 §3 #4.
5. **F4-связка с outbound HMAC-rotation (E14.6 §3).** При вводе
   SSRF-allowlist заодно ревью per-subscription secret rotation.

    Keycloak за WAN/HA-LB с холодным стартом — вынести в конфиг (§4).
