## 0. TL;DR за 30 секунд

- **Проблема.** SPEC §3.7 требует retention 7/10 лет для `audit_log`. Без
  партиций retention = `DELETE`, что запрещено append-only-триггерами +
  INSERT-only grant'ом (V070) и непредсказуемо рвало бы hash-chain (V072).
  Open question E14 round 1 §3 #3 («chain × partition drop») надо было решить.

- **Решение по chain (зафиксировано).** Цепочка остаётся **глобальной** (по
  `id ASC` через все партиции). Партицию **нельзя** дропать внутри retention-
  окна; дроп — только за его пределами И только после immutable-архива
  сегмента (round 10, S3). Форсится функцией
  `audit.drop_audit_partition_if_archived(part, archived, retention)` —
  прямой `DROP` партиции в обход неё запрещён политикой. `verify-chain`
  не меняется (`findChainRange ... ORDER BY id` сшивает партиции).

- **V073 (audit, auto-apply).** Идемпотентный guarded rebuild:
  `audit.audit_log` → `PARTITION BY RANGE (occurred_at)`, помесячные партиции
  на весь диапазон данных + DEFAULT (safety-net, INSERT никогда не падает),
  копия строк байт-в-байт (id/occurred_at/hash'и сохранены → chain цел),
  swap, пересозданы индексы/триггеры/grants V070, sequence detach+re-own.
  + `audit.ensure_audit_partition(month)` для cron.

- **Вынужденное изменение схемы.** Ключ партиции обязан входить в каждый
  UNIQUE/PK: PK `(id)`→`(id, occurred_at)`; UNIQUE `(event_id,event_type)`
  [V071] → `(event_id,event_type,occurred_at)`. `AuditLogDao.insert`
  `ON CONFLICT` обновлён синхронно — идемпотентность сохранена (`occurred_at`
  = `event.occurredAt()`, детерминирован для replay'я).

- **code_item — opt-in.** `bootstrap/sql/optional/code_item_partitioning.sql`
  (папка вне `bootstrap/sql/migrations` → Flyway не сканирует). LIST по
  version_id + DEFAULT (без рефакторинга кода) +
  `authoring.ensure_code_item_partition(version_id)` для выноса DEPRECATED-
  версий. На пилот не применяется.

- **Backend-only, без UI.**

## 1.0. ⚠️ Два pre-existing round-1 дефекта, вскрытых smoke'ом (исправлены)

Оба — в коде/миграциях E14 round 1 (коммит `a7625e9`), не пойманы т.к.
round-1 smoke никогда не гонялся. **Без обоих фиксов вся audit-схема (>= V072)
и round 7 не разворачиваются вообще.**

### Дефект A — V072 не применяется (permission denied)

Flyway ходит в БД под `rdmmesh_app` (он же owner таблиц). V070 делает
`REVOKE UPDATE,DELETE,TRUNCATE … FROM rdmmesh_app`. В PostgreSQL владелец
сохраняет ALTER/DROP, но **DML-привилегии у владельца тоже отзываются
REVOKE'ом**. V072 (hash-backfill) делает `UPDATE audit.audit_log …` → падает
на `ERROR: permission denied for table audit_log`, миграция rollback'ится,
сервис не стартует.

**Фикс (V072):** скобка `GRANT UPDATE … TO rdmmesh_app` … `REVOKE UPDATE …`
вокруг backfill'а, симметрично уже существующей `DISABLE…ENABLE TRIGGER`
(оба DO-блока guard'ятся `IF EXISTS rdmmesh_app`). Append-only-инвариант
восстановлен после backfill'а. Правка V072 in-place безопасна: миграция
**нигде успешно не применялась** (`flyway_schema_history` max success был
071), поэтому checksum-mismatch'а ни в одной среде нет.

### Дефект B — hash-chain не верифицируется (~50% строк)

`AuditChainHasher.formatOccurredAt` **усекает** до µs
(`truncatedTo(MICROS)`). `AuditService` хэшировал `event.occurredAt()`
(in-memory **нано**, напр. `.302652599` → усечение → `.302652`), а в колонку
`occurred_at timestamptz` PostgreSQL пишет с **округлением** до µs
(`.302653`). `verify` (и SQL-формула V072) пересчитывает из округлённой
колонки → `entry_hash` не сходится для всех строк, где 7-й знак даёт
округление вверх (≈половина). Headline-фича E14 (tamper-evident audit) была
сломана end-to-end.

**Фикс (`AuditService`):** усекаем `occurred_at` до µs **один раз**
(`event.occurredAt().truncatedTo(ChronoUnit.MICROS)`) и подаём это же
значение И в `computeEntryHash`, И в `dao.insert`. Тогда сохранённая колонка
== хэшируемое значение, Postgres уже-µs не округляет, verify сходится.
Минимальный фикс (выбор пользователя): `payload_canonical` по-прежнему
содержит nano-`occurred_at` внутри payload'а — но он хранится и
пересчитывается **verbatim** (одна и та же строка на write и verify),
поэтому на верификацию не влияет (косметика). Замороженная V072-формула
hash-input НЕ менялась; изменилось только то, какое значение времени в неё
подаётся (усечённое vs сырое нано).

**Валидация (реальный Postgres, §5):** до фикса `verify-chain`
`verified:false entry_hash mismatch`; после — `verified:true` на 4 событиях
4-eyes; tamper по-прежнему ловится (`verified:false, first_broken_at:3`).

> **Follow-up (не в minimal-scope):** (1) выровнять `payload_canonical`'s
> embedded `occurred_at` на µs для косметической консистентности; (2)
> testcontainers-IT на write→verify roundtrip (round 9) — чтобы класс
> «precision drift» не повторился; (3) задним числом пере-verify исторических
> подписей при ротации (E14.6 §3 #1) — отдельный round.

## 1. Что сделано

### 1.1. `bootstrap/sql/migrations/audit/V073__audit_log_range_partition.sql`

Auto-apply (Flyway сканирует `db/migration/audit`). Структура файла:

1. `audit.ensure_audit_partition(date) → text` — идемпотентно создаёт
   помесячную партицию `audit_log_yYYYYmMM`; проверяет, что таблица реально
   партиционирована.
2. `audit.drop_audit_partition_if_archived(text, boolean, interval) → void` —
   единственный легальный дроп: `RAISE` если `archived=false` либо верхняя
   граница внутри retention (default 10 лет). DEFAULT-партицию дропать
   запрещает.
3. `REVOKE EXECUTE … FROM PUBLIC` на обе (ops-функции, owner-only).
4. `DO $mig$` — guarded (relkind='p' → no-op; PITR/повторный прогон safe):
   create partitioned table → DEFAULT → помесячные на [min..max+1] → copy
   `ORDER BY id` → `setval` sequence → `OWNED BY NONE` → drop old → rename →
   `OWNED BY` re-own → пересоздать 4 индекса V070 → 3 append-only триггера
   (функция `audit.audit_log_no_modify` из V070 жива) → grants V070.

Вся мутация в одной tx (Flyway): сбой → полный rollback, таблица нетронута.

### 1.2. `rdmmesh-audit` — DAO + комментарии

- `AuditLogDao.insert`: `ON CONFLICT (event_id, event_type)` →
  `(event_id, event_type, occurred_at)`. Javadoc DAO + `AuditService`
  обновлены (ссылались на имя индекса V071 `audit_log_event_id_uq`, которого
  больше нет — констрейнт теперь в самой таблице).
- **Семантика idempotency не изменилась**: replay одного логического события
  несёт тот же `event_id`+`event_type`+`occurred_at` (occurred_at — это
  `event.occurredAt()`, не время вставки) → конфликт → `DO NOTHING`.

### 1.3. `bootstrap/sql/optional/code_item_partitioning.sql` (opt-in, НЕ миграция)

Папка `bootstrap/sql/optional/` не входит в resource-копирование
(`rdmmesh-app/pom.xml` копирует только `bootstrap/sql/migrations` →
`classpath:db/migration`), поэтому Flyway её не видит. Применяется вручную
(`psql -1 -f`) в maintenance-окне когда DEPRECATED-версии раздувают БД.
LIST по version_id + DEFAULT (поглощает всё → код не меняется) +
`authoring.ensure_code_item_partition(uuid)` (DETACH default → создать
партицию version_id → перенести строки → ATTACH default). Пересоздаёт
индексы V020 + триггеры V022/V023.

### 1.4. Runbooks

- `docs/runbooks/audit-log-retention.md` — chain-решение, ежемесячный cron,

## 3. Что осталось — follow-up'ы

1. **cron `ensure_audit_partition`.** Сейчас функция есть, но планировщик
   (cron/k8s CronJob под `rdmmesh_admin`) — задача эксплуатации (runbook §2).
   DEFAULT-партиция страхует от пропуска (данные не теряются, chain цел),
   но требует remediation. Не блокирует, но настроить до prod.
2. **Архив-перед-дропом (round 10, S3 immutable).** `drop_audit_partition_if_archived`
   требует `archived=true`, но *проверки факта* архива (lookup в S3-манифесте)
   нет — параметр передаёт оператор честно. Связать с round 10.
3. **code_item: guarded drop версии.** Шаблон даёт `ensure_code_item_partition`,
   но дроп выделенной партиции — ручной (нет функции-стража как у audit).
   Добавить при первом реальном применении (runbook §5).
4. **Testcontainers IT (round 9).** Smoke прогнан вручную (§5); чтобы класс
   «precision drift» (дефект B) и V072-permission (дефект A) не повторились —
   нужен автоматический testcontainers-IT на write→verify roundtrip +
   apply-all-migrations. Формально round 9.
6. **`payload_canonical` embedded `occurred_at` — nano, не µs.** Не ломает
   verify (хранится/пересчитывается verbatim), но косметически расходится с
   усечённой колонкой. Выровнять при удобном случае (не minimal-scope).
5. **Прямой DML по партиции.** STATEMENT-триггеры висят на родителе; прямой
   `DELETE FROM audit.audit_log_y2026m05` суперюзером их не тронет.
   Mitigation — REVOKE роли (как в V070, defense-in-depth). Документировано.

| Проверка | Результат |
