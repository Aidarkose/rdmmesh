## 0. TL;DR за 30 секунд

- **Backend (миграция V022):** четыре триггера на `authoring.code_item`:
  - **BEFORE INSERT/UPDATE OF parent_key:** `code_item_check_self_parent` —
    блокирует `parent_key = key_parts` с `check_violation`.
  - **AFTER INSERT:** `code_item_closure_on_insert` — 4 шага Bill Karwin
    pattern: self-reflexive → ancestors→NEW → NEW→existing_subtree →
    cross-product. Покрывает out-of-order вставки (parent после children).
  - **AFTER DELETE:** `code_item_closure_on_delete` — удаляет все строки
    closure где `ancestor=OLD OR descendant=OLD`.
  - **AFTER UPDATE OF parent_key:** `code_item_closure_on_move` — удаляет
    старые ancestor-связи subtree, добавляет cross-product
    `new_parent_chain × subtree`.
  - **Backfill** при первом применении V022: `TRUNCATE closure + WITH RECURSIVE`
    для всех существующих версий. Идемпотентно на свежей БД.
- **AuthoringService:** убраны все вызовы `CodeItemClosureDao.rebuild(versionId)`
  из `addItem` / `updateItem` / `deleteItem` / `bulkUpsertJson` / `bulkUpsertCsv`
  / `createDraft`. Триггеры делают всё incrementally.
  `CodeItemClosureDao.rebuild()` остался как **diagnostic / disaster-recovery
  API** — не вызывается в обычной работе.
- **Frontend `HierarchyTree`:**
  - Новый компонент `HierarchyTree.tsx`. Строит forest из `items[]` через
    `serializeKey(parts)=JSON.stringify(parts)`. Сортировка по `order_index`
    затем `key`. Композитные ключи поддерживаются (одинаковая сериализация
    для match'а `parent_key ↔ key_parts`).
  - **Drag-and-drop** только в DRAFT. На drop:
    - cycle-protection: запрещён drop в собственное subtree (walk-down проверка
      на frontend'е — backend всё равно заблокирует через `check_violation`).
    - drop **into** node → `parent_key = target.key_parts`.
    - drop **at gap** → `parent_key = target.parent_key` (становится sibling).
    - no-op detection: target = текущий parent — backend не дёргается.
  - Optimistic-lock через `expected_row_version`; 409 → message + invalidate.
- **VersionPage** получил `<Segmented>` Grid / Tree для CodeSet'ов с
  `hierarchy_mode != "NONE"`. Tree-view доступен и в read-mode (без drag),
  и в DRAFT (с drag-drop).
- **Bundle:** app-код 27.5 → **28.7 KB gzip** (+1.2 KB на HierarchyTree +
  Segmented). vendor-antd без изменений.

## 1. Что сделано

### 1.1. Миграция V022 — детально

**Файл:** `bootstrap/sql/migrations/authoring/V022__code_item_closure_triggers.sql`

#### BEFORE-trigger: запрет self-parent

```sql
CREATE FUNCTION authoring.code_item_check_self_parent() ...
  IF NEW.parent_key IS NOT NULL AND NEW.parent_key = NEW.key_parts THEN
      RAISE EXCEPTION 'code_item.parent_key cannot equal key_parts (self-reference): %'
        USING ERRCODE = 'check_violation';
  END IF;
```

Срабатывает на `BEFORE INSERT` и `BEFORE UPDATE OF parent_key`. ERRCODE
`check_violation` — клиент-side (`AuthoringService`) уже ловит это как
`IllegalArgumentException` через стандартный JDBI exception mapping
(`UnableToExecuteStatementException.isUniqueViolation` фильтрует только
unique; остальные DB-ошибки прокидываются наверх).

#### AFTER INSERT trigger

Четыре шага. Пример: имеем дерево A→B→C. Делаем `INSERT D` с `parent_key=B`:

1. **Self:** `(D, D, 0)`.
2. **Ancestors→NEW:** copy `(A,B,*)` ⇒ `(A,D,2)`; copy `(B,B,0)` ⇒ `(B,D,1)`.
3. **NEW→existing_subtree:** если уже есть `E` с `parent_key=D` — copy
   closure E ⇒ добавить `(D,E,*+1)`.
4. **Cross-product ancestors × existing_subtree:** `(A,E,2)`, `(B,E,1)` —
   соединяет ancestors `D` с descendants `D`.

Шаги 3-4 покрывают **out-of-order INSERT** в bulk: даже если children
вставлены до parent'а, после INSERT parent'а closure self-исправится.

#### AFTER DELETE trigger

```sql
DELETE FROM closure
 WHERE version_id = OLD.version_id
   AND (ancestor_key = OLD.key_parts OR descendant_key = OLD.key_parts);
```

После DELETE node X: если у X были дети — их `parent_key` в `code_item`
по-прежнему указывает на X (FK нет — `parent_key` это jsonb, не reference).
Application-level concern: висячие children отображаются в Tree-view как
orphan-roots. Чисто закрывает round 2; стандартный fix — каскадно удалять
subtree приложением.

#### AFTER UPDATE OF parent_key trigger (move)

```sql
-- 1. Снять старые ancestor-связи subtree
DELETE FROM closure
 WHERE descendant_key IN (subtree of NEW)
   AND ancestor_key  NOT IN (subtree of NEW);

-- 2. Добавить новую parent_chain × subtree (если new parent NOT NULL)
INSERT INTO closure
SELECT NEW.version_id, anc.ancestor_key, sub.descendant_key,
       anc.depth + sub.depth + 1
  FROM closure anc CROSS JOIN closure sub
 WHERE anc.descendant_key = NEW.parent_key
   AND sub.ancestor_key   = NEW.key_parts;
```

`NEW.parent_key = NULL` → узел становится корневым (шаг 2 пропускается).
`NEW.parent_key IS NOT DISTINCT FROM OLD.parent_key` → ранний return (no-op).

#### Backfill

```sql
DO $$ ...
    TRUNCATE TABLE authoring.code_item_closure;
    FOR v_id IN SELECT DISTINCT version_id FROM authoring.code_item LOOP
        INSERT ... WITH RECURSIVE walk AS (...);
    END LOOP;
END$$;
```

Гарантирует консистентность после миграции — даже если до V022 closure был
рассинхронизирован (теоретически возможно через ручной SQL или баг). На
свежей dev-БД (closure пустая) — no-op.

### 1.2. AuthoringService — removed rebuilds

| Метод | Было | Стало |
|---|---|---|
| `createDraft` (clone items из base) | INSERT items + `rebuild()` | INSERT items (триггеры на каждом INSERT) |
| `addItem` | INSERT + `rebuild()` | INSERT (AFTER-INSERT trigger) |
| `updateItem` | UPDATE + `rebuild()` | UPDATE (AFTER-UPDATE-OF-parent_key trigger; no-op если parent не изменился) |
| `deleteItem` | DELETE + `rebuild()` | DELETE (AFTER-DELETE trigger) |
| `bulkUpsertJson` / `bulkUpsertCsv` | N×INSERT/UPDATE + `rebuild()` | N×INSERT/UPDATE (N×trigger вместо O(N²)) |

`CodeItemClosureDao.rebuild()` остался — это **diagnostic API**, javadoc
обновлён ("E13 round 2 — disaster-recovery").

### 1.3. UI HierarchyTree

```
rdmmesh-ui/src/components/HierarchyTree.tsx
```

API:
```tsx
<HierarchyTree
  items={p.items}
  versionId={vid}
  codesetId={codeset.id}
  editable={status === "DRAFT"}
/>
```

**buildForest(items)** — O(N) построение `TreeNode[]`:
1. `Map<serializedKey, TreeNode>` — индекс по key.
2. Для каждого item — найти parent через `parent_key.serializedKey` в map.
3. Items без parent_key или с orphan parent_key — корни.
4. Сортировка `order_index` затем `key` (recursive).

**onDrop(info)** обрабатывает:
- `info.dragNode.key` — serialized JSON key of dragged item.
- `info.node.key` — target.
- `info.dropToGap` — true если drop между нодами, false если внутрь.

Walk-down cycle-check на frontend (preempt 409 от backend'а). Backend всё
равно блокирует через `code_item_check_self_parent` BEFORE-trigger
(только self-cycle X→X; глубокие циклы A→B→C→A backend пока не ловит —
см. §3 #2).

**Drag-drop visual:** AntD `<Tree draggable showLine blockNode>`. Hint
выводится при `editable=true`. `defaultExpandAll` — для пилотных объёмов
(<100 items) удобно; для крупных — V1+ переехать на virtual-scroll либо
fold-by-default.

### 1.4. VersionPage — Grid/Tree toggle

```tsx
{codeset.data && codeset.data.hierarchy_mode !== "NONE" && (
  <Segmented value={viewMode} onChange={...}
    options={[{label:"Таблица",value:"grid"},{label:"Дерево",value:"tree"}]} />

## 2. Контракт

### 2.1. Backend (без изменений REST)

PATCH `/api/v1/versions/{id}/items/{itemId}` — без изменений семантики, но
теперь `parent_key`-изменение **триггерит closure-update в той же
транзакции**. Если транзакция rollback'нется (например, optimistic-lock 409
сразу после move) — closure тоже откатится. ACID-консистентность сохранена.

### 2.2. Backend (новые ошибки)

`POST` / `PATCH` items могут вернуть **400 Bad Request** через
`IllegalArgumentException` (см. `CodeItemResource.create:97`,
`CodeItemResource.patch:135`) если backend получил
`parent_key = key_parts` — BEFORE-trigger выбросит `check_violation`,
JDBI обернёт в `UnableToExecuteStatementException`, service пробросит как
`IllegalArgumentException` → resource → 400.

### 2.3. Frontend

- Новый компонент `HierarchyTree` использует тот же `apiMutations.patchItem`
  что и `ItemsTable`. Те же query-keys инвалидируются.
- `Segmented` контролируется локальным state'ом `viewMode` (per page mount).
  Перезагрузка страницы — сбрасывает в `grid`. Это by-design (не сохраняем в
  localStorage, чтобы пользователь не залип в Tree-view на CodeSet'е, где
  parent_key редко нужен).

## 3. Что осталось доделать в E13 — мягкие follow-up'ы

Ничего не блокирует переход к E14. Список того, к чему стоит вернуться:

1. **Benchmark closure-rebuild на 10k items.** Reasonable next step — поднять
   `make up`, через bulk-import загрузить 10k items одной транзакцией,
   замерить latency `addItem` / `updateItem` / `move`. Ожидание:
   - INSERT: O(depth × subtree_size) на каждый item; для типичных deep<10 и
     subtree<100 — единицы ms.
   - UPDATE OF parent_key: O(subtree_size × ancestor_chain) — самый дорогой
     операционный путь.
   - Bulk import 10k items: ожидание < 30 сек (vs prior > 5 минут на rebuild).
2. **Cycle detection для глубоких циклов.** Сейчас blocked только
   `parent_key = key_parts` (self-cycle). Глубокий цикл `A→B→C→A` через
   несколько UPDATE'ов теоретически возможен:
   - UPDATE A SET parent_key=B (OK, B не в subtree A);
   - UPDATE B SET parent_key=C (OK, C не в subtree B);
   - UPDATE C SET parent_key=A — теоретически создаст цикл.

   Сейчас закрывается частично через cross-product trigger: попытка вставить
   `(A,A,>0)` будет отброшена ON CONFLICT (PK), но **closure станет
   неконсистентной** (отсутствуют некоторые ancestor-связи). Правильный fix
   — отдельная функция `authoring.detect_cycle(version_id) → boolean` +
   AFTER-UPDATE trigger, который её вызывает и `RAISE EXCEPTION` если цикл
   обнаружен. V14 (compliance hardening) или раньше.

3. **Sibling-reorder через drag-drop.** Сейчас `onDrop` обновляет только
   `parent_key`, не `order_index`. Если пользователь хочет переупорядочить
   siblings — это через inline-edit `order_index` поле. Drag-drop при
   `dropToGap=true` сейчас trait'ится как «сменить parent», но более
   интуитивно было бы «изменить позицию среди siblings». V1+ или round 2c.

4. **Tree virtualization для больших деревьев.** AntD `<Tree>` без virtual
   рендерит все узлы — для деревьев >1000 узлов это лагает. AntD умеет
   `virtual={true}` (DirectoryTree-mode). Перевести при первой жалобе.

5. **CROSS_CODESET (parent_ref) UI.** SPEC §3.4 поддерживает cross-codeset
   references через `parent_ref={codeset_id, key_parts}`. Сейчас Tree-view
   строится только по INTRA `parent_key`. CROSS_CODESET items отрисуются
   как изолированные узлы. UI cross-codeset reference editor — V1+.

6. **Closure rebuild как admin REST endpoint.** Сейчас `rebuild()` есть в
   DAO как Java-метод, но REST-вызова нет. Полезно для disaster-recovery:
   `POST /api/v1/versions/{id}/closure/rebuild` под `RDM_ADMIN`. Добавится
   когда понадобится.

7. **Тесты на триггеры.** Pure unit-тесты невозможны (это SQL). Integration
   через testcontainers (Postgres) — `@DropwizardAppRule` + сценарии
   `INSERT-out-of-order` / `move-to-root` / `move-cycle-block`. Мягкий debt,
   тот же что у E3/E4/E5/E10.

8. **LIST partitioning `code_item` по `version_id`.** SPEC §3.4 явно даёт это
   как pluggable («partitioning отключено в MVP, добавляется DDL-миграцией»).
   После V022 closure обслуживается incremental — partitioning не ломает
   логику (триггеры срабатывают per-row, индекс на partition-level). V14.

9. **Backward-compat — `code_set_version_id ON DELETE CASCADE` в closure.**
   `code_item_closure.version_id` REFERENCES `code_set_version`. При DELETE
   CodeSetVersion (например, `deleteDraft`) closure уйдёт каскадом
   (`AuthoringService.deleteDraft:172`). Триггер `code_item_closure_on_delete`
   при этом не сработает (мы делаем CASCADE delete не через row-level
   trigger, а constraint-level). Это правильно — закрытие cascading.

## 4. Технический долг и решения

| Что | Где | Когда снять / следующий шаг |
|---|---|---|
| `rebuild()` остался как diagnostic-API | `CodeItemClosureDao` | by-design; disaster-recovery |
| Self-cycle блокируется, глубокий цикл — нет | V022 BEFORE-trigger | V14 — отдельная detect_cycle function |
| Sibling reorder через order_index, не через drag | `HierarchyTree.onDrop` | V1+, сейчас drag = move |
| Tree без virtualization | `HierarchyTree` | AntD Tree `virtual={true}` если объёмы вырастут |
| Backfill TRUNCATE+rebuild при V022 | миграция | разово; на свежей БД no-op |
| Closure-table нагрузка при bulk import 10k items | trigger N×depth | benchmark в smoke (см. §3 #1) |
| CROSS_CODESET (parent_ref) не в Tree-view | `HierarchyTree.buildForest` | V1+ |
| `code_item_check_self_parent` trigger через RAISE EXCEPTION | V022 | возможно заменить на CHECK через `(parent_key IS NULL OR parent_key <> key_parts)` — JSONB-сравнение в CHECK легально |
| `parent_key` orphan reference (parent не существует) — в `HierarchyTree` рендерится как root | `HierarchyTree.buildForest` | by-design; в read-mode безопасно, в drag — orphan можно "усыновить" drop'ом |

# Проверка closure:
docker exec rdmmesh-postgres psql -c \
  "SELECT ancestor_key, descendant_key, depth FROM authoring.code_item_closure
    WHERE version_id='$V2' ORDER BY depth, ancestor_key::text;"
# Ожидание (после AFTER-INSERT trigger у DEPT — шаги 3+4 подцепляют existing DESK):
#  ancestor=DEPT descendant=DEPT depth=0
#  ancestor=DEPT descendant=DESK depth=1     ← добавлен step 3 при INSERT DEPT
#  ancestor=DESK descendant=DESK depth=0
```

#### Шаг 3 — move (UPDATE OF parent_key)

```bash
# Изначально: HQ → RETAIL → BRANCH_KZ. Move BRANCH_KZ под прямой HQ.
ITEM_ID=$(... id of BRANCH_KZ ...)
ROW_VER=0
curl -X PATCH -d "{\"expected_row_version\":$ROW_VER,\"parent_key\":[\"HQ\"]}" \
    .../versions/$V/items/$ITEM_ID

# Closure после move:
docker exec rdmmesh-postgres psql -c \
  "SELECT ancestor_key, descendant_key, depth FROM authoring.code_item_closure
    WHERE version_id='$V' ORDER BY depth, ancestor_key::text;"
# Ожидание:
#  - (RETAIL, BRANCH_KZ, 1) удалён (step 1 of move trigger)
#  - (HQ, BRANCH_KZ, 1) добавлен (step 2 — cross-product)
#  - (HQ, RETAIL, 1) остался
#  - (RETAIL, RETAIL, 0) и (BRANCH_KZ, BRANCH_KZ, 0) остались
```

## 7. Открытые вопросы (актуальны для команды банка)

Без изменений с E13 round 1, плюс:

1–26 — без изменений (см. `E13-bitemporal.md` §7).
27. **Cycle-detection scope.** Сейчас blocked только self-cycle (X→X).
    Глубокий цикл (A→B→C→A через несколько UPDATE'ов) может оставить closure
    неконсистентной. Решить с командой compliance: нужно ли блокировать на
    уровне БД (через detect_cycle function + AFTER-trigger) или достаточно
    application-level (`AuthoringService` walk-up на каждый UPDATE)?
28. **Sibling-reorder UX.** Drag-drop сейчас изменяет `parent_key`.
    Альтернатива: `dropToGap=true` интерпретировать как «изменить позицию
    среди siblings» (UPDATE order_index siblings). Решить с UX-аналитиками
    после первой работы с реальными Position/System данными.
29. **Tree virtualization threshold.** Сейчас `defaultExpandAll` рендерит
    весь forest. Когда переключаться на `virtual={true}` (свёрнутые узлы,
    rendering только видимых)? 1000 / 5000 узлов — порог?
30. **Disaster-recovery REST endpoint** для closure rebuild —
    `POST /api/v1/versions/{id}/closure/rebuild` под `RDM_ADMIN`?
    Нужен ли (на пилоте — нет) или V1+ feature?
