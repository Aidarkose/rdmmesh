## 0. TL;DR

- **`WorkflowGraph`** (workflow.internal) — топология как значение:
  рёбра `Edge(from,to)→EdgeSpec(action,Kind,reject,recordReviewer,
  setApprover)`. `Kind∈{SUBMIT,STEWARD,OWNER,SYSTEM}` определяет
  runtime-guard. `defaultFourEyes()` воспроизводит прежнюю enum-матрицу
  **1:1**.
- **`StateMachine.validate`** теперь graph-driven:
  `validate(req)` → `validate(req, defaultFourEyes())` (delegating —
  поведение E5 байт-в-байт, сообщения/исключения/Decision идентичны;
  StateMachineTest 22/22 зелёный, `resolveAction`/`nextRequiredRole`/
  `allowedTransitions` не тронуты). Новый публичный
  `validate(Request, WorkflowGraph)` — для произвольных графов.
- **`WorkflowGraphInvariants`** — статическая compliance-сеть (замена
  «no-bypass по построению», цена B по ADR-0010). Теорема: достичь
  `OWNER_APPROVED` нельзя, не пройдя STEWARD-approve, затем
  OWNER-approve-ребро ⇒ runtime-guard'ы STEWARD/OWNER ⇒ 3 разных лица.
  Инварианты: достижимость; на каждом простом пути DRAFT→OWNER_APPROVED
  STEWARD-ребро раньше OWNER-ребра; в терминал только OWNER-approve;
  reject — только STEWARD/OWNER.
- **Тесты (локальные, не нужен Docker):** WorkflowGraphInvariantsTest 9
  — дефолт compliant; обходные графы (skip-steward / fake-kind /
  submit-в-терминал / system-reject / недостижимость) отвергаются;
  self-approval на кастомном compliant-графе всё равно 409; parity
  `validate(req)`≡`validate(req,default)`.

## 2. Поведение / совместимость

- **Нулевое изменение** для всех текущих путей: enum-движок и Flowable
  (round 1/2) используют `validate(req)` → дефолтный граф = прежняя
  матрица. Сообщения/коды/Decision-флаги идентичны (StateMachineTest
  22/22 — регресс-доказательство).
- `WorkflowService` пока зовёт `validate(req)` (дефолт) — per-domain
  граф НЕ подключён (это B2). Т.е. функционально пользователь ещё не
  видит расходящихся маршрутов; B1 — включающий рефактор + страховка.

## 4. Что осталось (B2/B3 — ADR-0010 слайс-план)

1. **B2:** per-domain BPMN кодирует граф (flowable extension elements) →
   парс при деплое → `WorkflowGraphInvariants` как **deploy-time gate**
   (невалид → 400, расширить `BpmnTemplateValidator`) → хранить граф
   рядом с `workflow_template` (V032 +колонка/таблица) →
   `WorkflowService` выбирает граф домена версии (резолв как в
   `FlowableWorkflowEngine.resolveDomain`). Тогда домен реально получает
   свою топологию.
2. **B3:** security-review (OWASP + no-bypass на произвольных графах,
   как E14.8) + property-тесты перед prod-включением кастомных графов
   (DoD §5.4 gate).
3. CI-прогон FlowableWorkflowIT 3/3 (round 1/2 регресс — поведение
   дефолта не изменилось).
4. Без изменений: Helm-проверка на стенде (E16.3 §3), Q56/Q62-финал.
