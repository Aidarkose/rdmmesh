## 0. TL;DR

- **Item 1 — orphan cleanup (реализовано).** `VersionDeletedDomainEvent`
  (api.eventbus, hand-record как ClosureRebuild) эмитится
  `AuthoringService.deleteDraft(versionId, actor)` ПОСЛЕ commit'а delete-tx
  (best-effort, SyncEventBus изолирует подписчика — удаление не
  откатывается). При `engine=flowable` composition-root подписан →
  `FlowableEngineManager.cancelForVersion` гасит инстансы с
  businessKey=versionId (идемпотентно). `rdmmesh-audit` авто-логирует
  (`event_type=VERSION_DELETED`, aggregate=VERSION, actor) — append-only
  audit бонусом. IT: `orphanInstanceCancelledOnVersionDelete`.
- **Item 3 — Helm (реализовано, НЕ deploy-протестировано).**
  `helm/rdmmesh/` — Deployment/Service/ConfigMap/Ingress(CSP+HSTS
  E14.8 §3 #2)/2×CronJob(каркас по runbook'ам)/NOTES/README. Секрет
  чарт НЕ создаёт (Vault/SOPS, SPEC §3.7) — только ссылается
  (`secret.existingSecret`). `RDM_WORKFLOW_ENGINE` в ConfigMap.
  Migrations — README §Migrations (нет migrate-CLI; managed-rollout
  `autoMigrate=true` либо follow-up). Нет helm/k8s в среде → не
  lint/template/deploy-проверено (честно, README §Статус).
- **Item 2 — topology-as-data (НЕ реализовано, by design).** ADR-0010
  **Proposed**: опции B (параметризованная SM из графа) / C (кастомная
  топология + compliance-валидатор) / A (текущее). Рекомендация: A пока
  бизнес не потребует расходящиеся маршруты; тогда C (статическая
  compliance-сеть), не B. Реализация — отдельный эпик после явного
  решения + security-review.
