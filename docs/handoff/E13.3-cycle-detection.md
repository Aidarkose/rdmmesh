## 1. Что сделано

### 1.1. Миграция V023 — детально

**Файл:** `bootstrap/sql/migrations/authoring/V023__code_item_deep_cycle_detection.sql`

#### Быстрый BEFORE-check на UI/PATCH flow

```sql
CREATE FUNCTION code_item_check_move_no_cycle() RETURNS trigger AS $$
BEGIN
    IF NEW.parent_key IS NOT NULL
       AND NEW.parent_key IS DISTINCT FROM OLD.parent_key THEN
        IF EXISTS (
            SELECT 1 FROM closure
             WHERE version_id     = NEW.version_id
               AND ancestor_key   = NEW.key_parts
               AND descendant_key = NEW.parent_key
        ) THEN
            RAISE EXCEPTION 'parent_key % is a descendant of % ... cycle'
                USING ERRCODE = 'check_violation';
        END IF;
    END IF;
    RETURN NEW;
END;
$$;

CREATE TRIGGER ... BEFORE UPDATE OF parent_key ON code_item ...
```

**Логика:** closure до операции уже отражает все subtrees. Если новый
`parent_key` уже встречается как descendant нашего узла — операция создаст
цикл. ERRCODE `check_violation` → JDBI оборачивает в
`UnableToExecuteStatementException` → `AuthoringService` пробрасывает →
resource отдаёт **400 Bad Request** (тот же путь, что V022 self-parent
check).

#### DEFERRED CONSTRAINT TRIGGER на пост-транзакционный invariant

```sql
CREATE FUNCTION code_item_closure_no_cycle_invariant() RETURNS trigger AS $$
DECLARE cycle_row record;
BEGIN
    SELECT ancestor_key, depth INTO cycle_row
      FROM closure
     WHERE version_id = NEW.version_id
       AND ancestor_key = descendant_key
       AND depth > 0
     LIMIT 1;
    IF FOUND THEN
        RAISE EXCEPTION 'closure cycle detected ... ' USING ERRCODE = 'check_violation';
    END IF;
    RETURN NULL;
END;
$$;

CREATE CONSTRAINT TRIGGER ...
    AFTER INSERT OR UPDATE OF parent_key ON code_item
    DEFERRABLE INITIALLY DEFERRED
    FOR EACH ROW EXECUTE FUNCTION code_item_closure_no_cycle_invariant();
```

**Логика:** `INITIALLY DEFERRED` откладывает проверку до конца транзакции.
Если транзакция содержит несколько write'ов (например, bulk-import 1000
items) — invariant проверяется один раз. Любой `(X, X, depth>0)` row в
closure означает цикл (через INSERT trigger cross-product такая строка
появится, если входные данные содержат цикл).

**Цена:** `O(1)` для типичной транзакции (один индекс-lookup). Для bulk
с N transitions — тоже один проход в конце, не N×check.

**Edge case:** триггер `RETURN NULL` потому что constraint triggers
обязаны это делать (Postgres doc); возврат NEW игнорируется в constraint
context.

#### Соотношение с V022

| V022 trigger | V023 trigger | Различие |
|---|---|---|
| `code_item_check_self_parent` (BEFORE) | — | Прямая проверка `NEW.parent_key = NEW.key_parts` без обращения к closure. Самый частый случай. |
| — | `code_item_check_move_no_cycle` (BEFORE) | closure-lookup, новый parent ∈ subtree(NEW). Покрывает 2+ -hop циклы при UPDATE. |
| — | `code_item_closure_no_cycle_invariant` (AFTER DEFERRED) | Защита от out-of-order bulk-import циркулярных данных. |

V022 и V023 не конфликтуют — defense-in-depth.

#### Sanity check на apply

```sql
DO $$
BEGIN
    SELECT count(*) ... WHERE ancestor_key = descendant_key AND depth > 0;
    IF bad_count > 0 THEN
        RAISE WARNING 'V023: existing closure contains % cycle-row(s). ...';
    END IF;
END$$;
```

На свежей dev-БД (closure пустая или согласованная) — no-op. На prod, если
miграция V022 backfill оставил рассинхронизированные данные — пользователь
увидит WARN в Flyway-логе с указанием на admin rebuild endpoint.

### 1.2. Backend service + REST

**`AuthoringService.rebuildClosure(versionId, admin)`** (handoff E13.2 §1.2 +
этот раунд):
```java
return jdbi.inTransaction(handle -> {
    VersionRow version = handle.attach(CodeSetVersionDao.class).findById(versionId)
            .orElseThrow(() -> new IllegalArgumentException("Version not found: ..."));
    CodeItemClosureDao dao = handle.attach(CodeItemClosureDao.class);
    int removed = dao.deleteAllForVersion(versionId);
    int inserted = dao.rebuild(versionId);
    int total = dao.countForVersion(versionId);
    log.warn("authoring: closure rebuild ... by_admin={}", admin);
    return new ClosureRebuildResult(versionId, removed, inserted, total);
});
```

`log.warn` намеренно — это disaster-recovery, должно быть видно в access-log'ах
и в централизованном log aggregator'е.

## 2. Контракт

### 2.1. Новое в round 3

| Endpoint | Метод | Используется в | Auth |
|---|---|---|---|
| `/api/v1/versions/{id}/closure/rebuild` | POST | RebuildClosureButton | `RDM_ADMIN` |

Response — `ClosureRebuildResult`:
```json
{
  "versionId": "<uuid>",
  "removed": <int>,    // строк удалено перед rebuild'ом
  "inserted": <int>,   // строк вставлено через WITH RECURSIVE
  "total": <int>       // итоговый count в closure для этой версии
}
```

**Errors:**
- `403` — нет роли `RDM_ADMIN`.
- `404` — версии не существует (`IllegalArgumentException` от service'а).
- `400` — versionId не UUID.
- `500` — неожиданная DB-ошибка (например, BD rejected `WITH RECURSIVE` через
  storage limit).

### 2.2. Новые backend-ошибки от V023 (через V022/V023 triggers)

PATCH `/versions/{id}/items/{itemId}` с `parent_key` (move) теперь может
вернуть **400 Bad Request** дополнительно к существующим случаям:

| Сценарий | Trigger | Сообщение (примерное) |
|---|---|---|
| `parent_key = key_parts` | V022 `code_item_check_self_parent` | `code_item.parent_key cannot equal key_parts (self-reference): ...` |
| `parent_key` ∈ subtree(item) | V023 `code_item_check_move_no_cycle` | `parent_key ... is a descendant of ... in version ... — would create cycle` |
| Out-of-order bulk создал цикл | V023 `..._no_cycle_invariant` (DEFERRED) | `closure cycle detected in version ...: key ... is its own ancestor at depth ...` |

Все три приходят как `IllegalArgumentException` → 400.

   деревьях >1000 узлов. AntD `<Tree virtual>` решает. Перевести при первой

   строится только по INTRA `parent_key`. CROSS_CODESET items отображаются
   изолированными узлами. UI cross-codeset editor — V1+.

4. **Benchmark closure-rebuild на 10k items** (handoff E13.2 §3 #1). После V023
   trigger DEFERRED invariant добавляет ещё один проход в конце транзакции —
   benchmark должен подтвердить, что overhead не превышает порога. Метрика:
   bulk-import 10k items < 30 сек.

5. **Testcontainers IT для триггеров V022 + V023.** Сейчас покрытие — backend
   `mvn verify` (unit + ArchUnit) и smoke с реальной БД. Integration test с
   Postgres testcontainer + сценарии:
   - INSERT out-of-order corrected
   - move to descendant blocked by V023 BEFORE
   - bulk circular input blocked by V023 DEFERRED
   - rebuild restores correct state after artificial corruption

   Тот же мягкий debt, что у E3/E4/E5/E10. Closes к E14 либо отдельным мини-PR'ом.

6. **`AuthoringService.rebuildClosure` audit-event.** Сейчас disaster-recovery
   только в `log.warn`. Для compliance желательно публиковать
   `ClosureRebuildDomainEvent` в EventBus → audit подхватывает. V14 territory.

7. **LIST partitioning `code_item`** по `version_id` (SPEC §3.4). После V022/V023
   trigger maintain'ит closure incremental — partitioning code_item не
   ломает логику (триггеры срабатывают per-row). V14.

## 4. Технический долг и решения

| Что | Где | Когда снять / следующий шаг |
|---|---|---|
| V022 `code_item_check_self_parent` остался хоть V023 покрывает | V022 + V023 | Defense-in-depth; не снимать — быстрее одного triggers, более явное сообщение об ошибке |
